# 小时报额度消耗预测规划

## 状态

- 状态：未开始
- 优先级：中
- 创建时间：2026-05-29
- 完成时间：-

## 背景

小时报目前可以展示最近一次额度快照、401 账号分析和分时明细。后续可以考虑增加一个简短的“消耗预测”块，用来回答：

- 下一小时预计会消耗多少 5h / 7d 额度。
- 按当前趋势大约什么时候会用完。
- 未来一段时间内是否有 5h 额度恢复，可以抵消一部分消耗。

这个功能只作为辅助判断，不作为告警触发依据。

## 约束

- 只读取本地 SQLite 里已经采集到的历史快照。
- 不请求 CPA，不调用 `collect`，不增加采集频率。
- 不引入并发查询或额外账号查询。
- 不新增敏感数据落库；优先使用现有快照中的时间、剩余额度、恢复时间。
- 小时报保持简短，预测信息不能让报告图片明显变长。

## 预测口径

优先使用更接近当前使用场景的历史样本：

1. 同小时 + 同工作日/周末样本。
2. 样本不足时，回退最近 6 小时趋势。
3. 仍然不足时，显示“历史数据不足，暂不预测”。

预测主指标优先使用总 5h 剩余额度，因为它更适合判断短周期风险。7d 额度可以作为辅助信息，不作为主要用完时间依据。

需要注意午饭、下班、晚上、周末等时段消耗会明显变化，不能简单拿全天平均值线性外推。

## 展示口径

只在小时报简短展示，不放进完整总结报。

建议展示内容：

- 下一小时预计消耗：例如 `预计 -6.5%`
- 预计用完时间：例如 `约 8 小时后，预计 21:30`
- 未来 1 小时恢复影响：例如 `预计恢复 +3.0%`
- 风险等级：`正常 / 关注 / 紧张 / 历史不足`

不展示每个账号的预测明细，避免小时报过长。

## 暂不实现原因

当前历史数据还不够稳定，且消耗受高峰期、低谷期、周末影响较大。过早实现容易给出误导性预测。

建议先积累一段时间的小时快照，再根据真实数据观察：

- 工作日和周末差异。
- 午饭、下班、夜间消耗差异。
- 5h 恢复事件对总额度的抵消效果。
- 401 或异常快照对预测样本的干扰。

## 后续测试方向

实现时至少覆盖这些场景：

- 同小时 + 工作日/周末样本优先。
- 周末样本不混入工作日预测。
- 同时段样本不足时回退最近趋势。
- 历史样本不足时不预测。
- 401、其他错误、无额度字段的快照不参与消耗样本。
- 结合 5h 恢复事件后，预计用完时间会被适当延后。

## 二阶段：容量建议

### 状态

- 状态：未开始
- 优先级：中
- 创建时间：2026-05-29
- 完成时间：-

### 背景

基础预测只能告诉用户“额度可能什么时候不够”。二阶段希望进一步给出可执行建议：如果预测发现额度撑不到下一轮高峰期，就估算至少需要补充多少个账号。

示例展示：

```text
预计到下一轮高峰期前缺口 28.00%，当前单账号平均可贡献 11.00%，建议至少补充 3 个账号。
```

### 高峰期口径

高峰期使用历史自动识别，不在配置里固定写死。

识别思路：

1. 从本地历史快照中按小时统计消耗强度。
2. 区分工作日和周末，避免周末低谷影响工作日预测。
3. 找出未来一段时间内预计消耗最高的小时段，作为下一轮高峰期。
4. 如果历史样本不足，显示“历史高峰样本不足，暂不给补号建议”。

### 建议形式

小时报中只展示“建议补充账号数 + 计算依据”，不展示复杂明细。

建议字段：

- 预计到高峰期前的总 5h 额度缺口。
- 当前单账号平均可贡献的总 5h 额度。
- 建议至少补充账号数。

### 计算思路

核心公式：

```text
建议补充账号数 = ceil(预计额度缺口 / 单账号平均可贡献额度)
```

其中：

- 预计额度缺口：预测到下一轮高峰期前，总 5h 额度低于安全线的部分。
- 单账号平均可贡献额度：按当前可用账号的总 5h 平均贡献估算。
- 计算结果向上取整，避免刚好不够。

### 展示范围

- 只在小时报预测块中展示。
- 完整总结报默认不展示，避免报告过长。
- 该建议只做参考，不自动触发采集、告警或账号操作。

### 约束

- 只读取本地 SQLite 历史快照。
- 不请求 CPA。
- 不增加采集频率。
- 不自动添加、启用、禁用账号。
- 401、其他错误、无额度字段的快照不参与样本计算。

### 后续测试方向

实现时至少覆盖这些场景：

- 能按小时识别历史高峰。
- 工作日和周末高峰分开统计。
- 历史样本不足时不输出补号建议。
- 额度足够撑到高峰期时不建议补号。
- 额度不足时按缺口和单账号贡献向上取整。
- 401、其他错误、无额度字段快照会被过滤。
